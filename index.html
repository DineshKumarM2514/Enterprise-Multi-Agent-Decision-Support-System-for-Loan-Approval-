Enterprise Multi-Agent Decision Support System for Loan Approval

A design report: six specialised agents — orchestrator, document verification, credit assessment, risk assessment, loan decision, and human review — coordinated by LangGraph, served on self-hosted open-source models via Ollama, built to run in production at a scale of 1M applications a year.

01

Executive Summary

Manual underwriting today is five people, an inbox, and no shared rubric. This report proposes an enterprise multi-agent decision-support system — five specialist agents plus one orchestrator, mirroring the real underwriting team, coordinated by LangGraph, served on self-hosted open-source models via Ollama. Humans stay in the loop for every low-confidence or flagged case; the agents just remove the repetitive 80%.

Business Need & Problem Statement

2.1  Where the current process breaks down

  • Latency — 3–5 people, 3–7 days, even with clean paperwork.
  • Inconsistency — no shared rubric; verdicts vary by reviewer.
  • Thin auditability — "why" means calling the underwriter.
  • Fraud & cost — manual cross-checks miss what structured checks catch, and burn hours on obvious cases.

2.3  Why multi-agent, not one large agent

Underwriting is four disciplines — document forensics, credit interpretation, risk modelling, policy decisioning — not one skill wearing four hats. One monolithic agent conflates them:

Failure mode of a single agentHow separation into specialist agents fixes it
Context overload — one giant prompt mixing OCR text, bureau JSON, policy rules, and reasoning instructions degrades reliability.Each agent gets a narrow, well-scoped context: only what it needs to do its one job.
No accountability — if the final decision is wrong, which reasoning step caused it?Every stage emits its own structured, evidence-cited output — the faulty step is directly visible in the trace.
Guardrails must be generic, or the whole prompt gets one blunt filter.Guardrails are targeted per agent (e.g. numeric-bounds checks only matter for risk/credit, forgery signals only for documents).
No natural seam to insert a human — either everything is manual or nothing is.The orchestrator can route just one stage's output to a human without re-running or blocking the rest.
Cannot scale or upgrade one capability independently.Each agent is its own service — the credit agent can move to a bigger model or scale out without touching the others.

Framing: the agent boundaries were chosen to mirror the org chart of a real underwriting team, not the other way around — that's what makes the escalation logic and the audit trail legible to compliance and to human reviewers who will work alongside the system.

The technical design above exists to serve this: faster decisions win customers, and the unit economics below are what makes the platform worth building as a product, not just a pilot.

2.4  Why this, why now

Speed wins the customer.

Scale without headcount.

5.7  Confidence & escalation, concretely

final_confidence = min(
    model_self_reported_confidence,   # agent's own calibrated score
    rule_based_confidence,            # deterministic penalties (data completeness, cross-source agreement)
)

escalate_to_human = (
    final_confidence < 0.75
    OR fraud_signal_detected
    OR sanctions_or_watchlist_hit
    OR requested_amount > policy.auto_approve_ceiling
    OR policy_exception_required
)

Confidence is deliberately the minimum of two independent estimates, not an average — a single weak signal should pull the whole decision toward human review rather than being diluted out.

6.5  Pipeline rule

Ingest → OCR/parse → normalize → feature store. Agents read only the feature store, never raw files. Keeps prompts small, PII exposure centrally controlled, and the pipeline replayable without re-running OCR (see Fig. 4).

Orchestration Layer

Agents never call each other directly. Every read and write goes through one shared, typed, versioned state object owned by the orchestrator — this is what keeps the audit trail complete and lets any single agent be replaced without touching the others.

8.1  Shared state shape

class LoanState(TypedDict):
    application_id: str
    verified_fields: dict | None        # written by doc_verification
    doc_confidence: float | None
    credit_profile: dict | None         # written by credit_assessment
    credit_confidence: float | None
    risk_profile: dict | None           # written by risk_assessment
    risk_confidence: float | None
    decision: dict | None               # written by loan_decision
    decision_confidence: float | None
    hard_stop_flags: list[str]
    escalated: bool
    trace_id: str

graph = StateGraph(LoanState)
graph.add_node("doc_verification", doc_verification_node)
graph.add_node("credit_assessment", credit_assessment_node)
graph.add_node("risk_assessment", risk_assessment_node)
graph.add_node("loan_decision", loan_decision_node)
graph.add_node("human_review", human_review_node)

graph.add_edge("doc_verification", "credit_assessment")
graph.add_edge("credit_assessment", "risk_assessment")
graph.add_edge("risk_assessment", "loan_decision")
graph.add_conditional_edges(
    "loan_decision",
    lambda s: "human_review" if needs_escalation(s) else END,
)
graph.add_edge("human_review", END)

app = graph.compile(checkpointer=PostgresSaver(conn))

8.2  Task routing & state management

  • Single writer per field — no two agents race on the same key.
  • Checkpoint every node — a crash resumes, it doesn't restart.
  • Retry once, then escalate — schema failure or timeout (20s SLA) both follow this path.

8.3  Inter-agent communication protocol

Read LoanState → call tools/model → return partial update → orchestrator validates + merges → publish to loan.state.events (Kafka). That one topic feeds both audit_log and observability — one event stream, not two systems kept in sync by hand.

09

Guardrails

Every single agent call — not just the final decision — passes through the same guardrail pipeline. Nothing an agent produces reaches shared state unfiltered.

Fig. 7 — Guardrail pipeline. This wraps every agent call in the graph — the diagram in Fig. 3 is what happens inside each of the "node" boxes.

9.1  Input guardrails

  • PII redact (Presidio) before anything is logged or prompted.
  • Injection scan — covers OCR'd text too (hidden PDF instructions).
  • File checks — type, size, malware, before OCR.

9.2  Output guardrails

  • Schema check — invalid JSON is rejected, not patched.
  • Citation match — cited numbers diffed against source rows; mismatch = hard reject.
  • Content classifier (Llama Guard) + numeric bounds at the type level.

9.3  Business-rule guardrails

Deterministic, git-versioned OPA policy, not LLM judgment. Runs after every Loan Decision output; can only downgrade or veto (DTI cap, no protected attributes, mandatory review above a threshold) — never upgrade past what policy allows.

9.4  Human-in-the-loop thresholds

Escalate when: confidence < 0.75 · fraud / watchlist flag · amount above auto-approve ceiling · thin credit file · agents disagree.

9.5  Audit & explainability

Every decision — automated or human — keeps its full reasoning chain, citations, and fired guardrails in the immutable audit_log, retained 7 years. Nothing is deleted or overwritten; corrections are new rows.

Security & Privacy

The trust boundaries below are the reason this system is built on self-hosted Ollama rather than a third-party model API: applicant PII never has to leave a VPC the organisation controls, at any stage, including inference.

Fig. 8 — Trust zones. PII is tokenised at the Application/Model boundary — the LLM never sees a raw PAN, phone number, or address, only reference tokens it can't resolve.

10.1  Data privacy

  • AES-256 (KMS keys) on SSN/PAN/phone/address, at rest and in objects.
  • TLS 1.3 external, mTLS internal.
  • Tokenised before the LLM — raw PII never enters model context.

10.2  Access control

Keycloak, short-lived OIDC tokens. Roles: underwriter · senior_underwriter · auditor · admin · applicant (read-only). Every service account is least-privilege.

10.3  Regulatory compliance

Fair-lending (no protected attributes, quarterly disparate-impact review) · data-subject rights (access/correction/erasure via the audit pipeline) · SOC 2–style change management.

10.4  Model security

Ollama never exposed publicly; per-request context isolation, gateway rate limiting, and a red-team suite (injection, jailbreak, exfiltration) before every model-version promotion.

11

MLOps / DevAgentOps Architecture

Prompts and models are versioned artifacts, not editable text — every change goes through the same gate before it can touch a real application.

Fig. 9 — Prompt/model change pipeline. Nothing skips the Eval Harness gate — not even a one-line prompt wording change.

11.1  Model lifecycle

Golden eval set of outcome-known applications, held out. Prompts live in git; every change re-runs the eval harness (accuracy, calibration / Brier score, guardrail trigger rate). Model Registry pins exact Ollama tag + quantisation per agent, per environment.

11.2  CI/CD for agents

PR → lint + schema check → unit tests (mocked LLM) → eval harness → staging → shadow → canary 5% → full rollout. Any regression blocks promotion; rollback is a registry pin change, not a redeploy.

11.3  Observability

Langfuse traces every call (prompt, output, latency, cost). Prometheus/Grafana track p50/p95 latency, escalation rate, and guardrail trigger rate, with drift alerting.

11.4  Feedback loop

Every human verdict becomes a labelled example. Disagreement cases are reviewed weekly, prioritised into the next eval set, and periodically into a LoRA fine-tuning pass.

Deployment & Infrastructure

One Kubernetes cluster, one microservice per agent, one GPU node pool for inference, and a stateful-services tier that can run self-hosted end to end.

Fig. 10 — Everything inside the dashed boundary is deployable via one Helm umbrella chart; the GPU pool is the only part that needs specialised nodes.

12.1  Microservices

api-gateway · orchestrator-svc · doc-verification-svc · credit-assessment-svc · risk-assessment-svc · decision-svc · human-review-svc (+ UI) · guardrails-svc · audit-svc · ollama-inference-pool

12.2  Kubernetes & IaC

One Deployment + HPA per service. Ollama is a StatefulSet on a tainted GPU node pool, so CPU services never compete for GPU scheduling. Helm per service, Terraform for VPC/KMS/Postgres (or Patroni for fully self-hosted).

12.3  Open-source-first hosting

The full stack runs self-hosted on-prem or in a private VPC on any cloud — predictable cost, full data residency. Inference can burst to a rented GPU cloud at peak without ever routing data through a closed API.

12.4  API gateway & mesh

Kong/APISIX at the edge, NGINX/Envoy ingress. A service mesh (Istio/Linkerd) is worth adding past ~8–10 services for uniform mTLS/retries — deliberately not day-one.

13.1  Current baseline (pilot)

500–2,000 applications/month: one orchestrator replica, one pod per agent, one Postgres, one small Ollama node. Every scaling technique below is added only when metrics say it's needed.

13.2  Where it breaks first

LLM inference is GPU-bound — the first bottleneck, not the database. Second: synchronous request/response couples client latency to full-pipeline latency, so spikes degrade UX instead of just queueing.

Fig. 11 — Same logical pipeline as Fig. 2, redrawn for scale: synchronous request/response becomes queue-fed, autoscaled worker pools.

13.3  Horizontal scaling strategy

  • Async + queued — 202 Accepted, KEDA scales each agent on its own queue depth.
  • Replicated, right-sized inference — autoscaled GPU pool; small models for extraction, big model only for decision.
  • Cache + partition data — Redis for repeat bureau/employer lookups; monthly-partitioned Postgres tables.
  • Stateless everywhere — checkpointed state means any replica resumes any application.

Testing & Evaluation Strategy

14.1  Per-agent evaluation

Each agent graded on its own golden set — extraction accuracy, score-band accuracy, flag precision/recall — plus a Brier score for confidence calibration.

14.2  End-to-end evaluation

Full pipeline run against held-out, outcome-known applications: outcome accuracy and agreement rate vs. the human underwriters who originally decided them.

14.3  Shadow mode → canary → full rollout

Shadow mode alongside the manual process for several weeks (no effect on applicants) → canary 5% (human-reviewed safety net) → 25% → full, with automatic rollback if agreement rate drops.

Model drift when a new open-source model version is adoptedEval harness gate in CI blocks any promotion that regresses golden-set accuracy or calibration (§11.2)
GPU capacity shortfall at peak demandBurst-to-cloud GPU strategy, per-agent model right-sizing (§13.3)

Phased Implementation Roadmap

Phase 0

Discovery & data readiness

Curate the golden evaluation set, map current policy rules into OPA-codifiable form, confirm document/bureau/employer data access.

Phase 1

Pilot — 3 core agents, shadow mode

Document Verification, Credit Assessment, Risk Assessment running in shadow alongside manual process; no real decisions affected.

Phase 2

Decision agent + guardrails hardening

Add Loan Decision and Human Review agents; land the full guardrail pipeline (§9) and security boundary (§10) before any canary traffic.

Phase 3

Canary → production cutover

5% → 25% → 100% real traffic with rollback triggers on agreement-rate regression; underwriter console live for escalations.

Phase 4

Scale-out to the 1M/yr target

Introduce async queueing, KEDA autoscaling, read replicas, and model right-sizing as §13 metrics warrant — not before.

Phase 5

Continuous improvement

Weekly disagreement review, periodic fine-tuning, quarterly fairness audit, ongoing eval-set expansion.
